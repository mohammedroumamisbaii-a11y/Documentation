The BI Framework

This documentation provides a 360-degree view of implementing a Business Intelligence layer into a multi-tenant ecosystem. We cover extraction, persistence, and secure delivery.

01. The Global Architecture

Before writing code, we must understand the data flow. A professional BI setup follows the E-L-V (Extract, Load, Visualize) pattern.

In this model, the BI engine acts as a separate microservice that "listens" to your database and serves visual components back to your main application via secure tunnels.

02. Production-Ready Deployment

Using Docker for orchestration ensures that the BI engine has its own isolated memory and process space.

Docker-Compose.yml 
version: '3.9'
services:
  bi_engine:
    image: metabase/metabase:latest
    container_name: metabase_instance
    ports:
      - "3000:3000"
    environment:
      - MB_DB_TYPE=postgres
      - MB_DB_DBNAME=metabaseappdb
      - MB_DB_PORT=5432
      - MB_DB_USER=db_admin
      - MB_DB_PASS=secure_password
      - MB_DB_HOST=postgres_host
    restart: always

03. The Security Protocol (JWT)

We never use public links in a commercial app. We implement Interactive Signed Embedding. This ensures Client A cannot see Client B's data.

The Payload Logic

{
  "resource": { "dashboard": 1 },
  "params": {
    "organization_id": 502  // Locked parameter: The user only sees data for Org 502
  },
  "exp": 1712416800 // Token expires in 15 minutes
}

Note: The signature is created using the HS256 algorithm with your Private Embedding Key.

04. Frontend Component Delivery

The final integration involves rendering the signed URL. Here is how it looks in a generic modern component:

BI-Dashboard.component 
<section class="analytics-wrapper">
  <!-- Dynamic iframe with Signed URL from Backend -->
  <iframe
    src="https://analytics.your-domain.com/embed/dashboard/YOUR_JWT_HERE#bordered=true"
    frameborder="0"
    width="100%"
    height="1000px"
    allowtransparency
  ></iframe>
</section>
